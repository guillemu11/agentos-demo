# Entry Node Description Modal — Design Spec

**Date:** 2026-04-16  
**Branch:** feat/journey-email-builder-modal  
**Status:** Approved

## Overview

When a user clicks the Entry node on the Journey Builder canvas, a modal opens with two editable tabs:

1. **Description** — plain-English explanation of the audience segmentation, auto-generated by Claude from the SQL the first time the modal opens.
2. **SQL** — the raw SQL query, directly editable.

Both tabs are editable and saved via a direct PATCH to the DSL (no agent involvement).

---

## 1. DSL Change

Add an optional `description` field to `entry.source`:

```json
"entry": {
  "source": {
    "type": "master_de_query",
    "master_de_key": "MasterDE_Emirates",
    "target_de_name": "Journey_UAE_Gold",
    "sql": "SELECT SubscriberKey FROM [MasterDE_Emirates] WHERE market = 'UAE' AND tier IN ('Gold','Platinum') AND last_open_days > 90",
    "description": "Targets UAE-market Gold and Platinum tier members who have not opened any email in the past 90 days."
  }
}
```

- `description` is optional and free-text — not validated by `validateEntry`.
- `setEntrySource` mutator in `mutators.js` accepts `description` as an optional fourth parameter and includes it in the returned DSL.
- `dsl-schema.js` `validateEntry` does not need to change.

---

## 2. Backend — Two New Endpoints in `server.js`

### `POST /journeys/:id/entry/describe`

Generates a plain-English description of the entry segmentation using Claude.

**Request body:**
```json
{ "sql": "...", "master_de_key": "...", "target_de_name": "..." }
```

**Logic:**
- Auth check (session required).
- Calls Claude (`claude-sonnet-4-6`) with a prompt that instructs it to explain the SQL in 2–3 sentences of plain English for a marketing manager, mentioning audience criteria, market, tiers, and any time-based filters visible in the SQL.
- Returns `{ description: "..." }`.

**Error handling:** 400 if `sql` is missing; 500 on Claude failure with `{ error: message }`.

---

### `PATCH /journeys/:id/entry`

Saves edited description and/or SQL back to the DSL.

**Request body:**
```json
{ "description": "...", "sql": "..." }
```

**Logic:**
- Auth check.
- Fetch current journey DSL from DB.
- Apply changes: update `entry.source.description` and `entry.source.sql`, preserving `master_de_key` and `target_de_name`.
- Run `validateDsl` on the updated DSL.
- If invalid: return `400 { errors: [...] }`.
- If valid: save to DB, return `{ dsl }`.

## 3. Frontend

### 3a. `packages/core/journey-builder/mutators.js` — `setEntrySource`

Accept optional `description` parameter and include it in the returned source object:

```js
export function setEntrySource(dsl, { master_de, sql, target_de_name, description }) {
  return {
    ...dsl,
    entry: {
      source: {
        type: 'master_de_query',
        master_de_key: master_de,
        sql,
        target_de_name,
        ...(description !== undefined && { description }),
      },
    },
  };
}
```

### 3b. New Component: `EntrySourceModal.jsx`

**Location:** `apps/dashboard/src/components/journey/EntrySourceModal.jsx`

**CSS class prefix:** `.esm` (entry source modal), following the `.ebm` pattern.

**State:**
- `activeTab`: `'description' | 'sql'`
- `description`: string (from DSL or generated)
- `sql`: string (from DSL)
- `generating`: boolean (true while calling `/entry/describe`)
- `saving`: boolean (true while calling PATCH)
- `error`: string | null
- `validationErrors`: string[] (SQL validation errors from PATCH response)

**Behavior on open:**
- Initialize `sql` and `description` from `dsl.entry.source`.
- If `description` is empty/falsy → automatically call `POST /journeys/:id/entry/describe` and set `generating = true`. Show spinner in Description tab while loading.
- If `description` already exists → show it immediately, no network call.

**Tabs:**
- `Description` tab: `<textarea>` editable. Shows spinner overlay if `generating`.
- `SQL` tab: `<textarea>` with monospace font. Shows inline warning badge if SQL doesn't start with `SELECT` (client-side only, non-blocking).

**Footer:**
- `Cancel` button → closes without saving.
- `Save` button → disabled while `generating || saving`. Calls `PATCH /journeys/:id/entry`. On success: calls `onSaved(newDsl)` and closes. On error: shows error message.

**Escape key:** closes modal (no confirmation needed since it's an edit, not a generation).

### 3c. `JourneyBuilderPage.jsx`

Add state:
```js
const [entryModalOpen, setEntryModalOpen] = useState(false);
```

Extend `handleNodeClick`:
```js
const handleNodeClick = (node) => {
  if (node.type === 'entry') {
    setEntryModalOpen(true);
    return;
  }
  if (node.data?.activity?.type === 'email_send' && !node.data.activity.mc_email_id) {
    setEmailBuilderActivity(node.data.activity);
  }
};
```

Add `<EntrySourceModal>` to JSX:
```jsx
<EntrySourceModal
  open={entryModalOpen}
  journeyId={id}
  dsl={dsl}
  onClose={() => setEntryModalOpen(false)}
  onSaved={(newDsl) => { setDsl(newDsl); setEntryModalOpen(false); }}
/>
```

### 3d. `EntryNode.jsx`

Add CSS class `journey-node--clickable` to the root `<div>` (sets `cursor: pointer`). No other changes needed.

### 3e. `index.css`

New `.esm` block (mirroring `.ebm` structure):
- Fullscreen overlay with scrim
- Panel with header, tab bar, tab content area, footer
- Description textarea: full height, resizable vertically
- SQL textarea: monospace font, full height, `font-size: 12px`
- Tab bar: two tabs with active indicator
- Generating spinner overlay on Description tab
- SQL warning badge: small red chip under textarea if SQL doesn't start with `SELECT`

### 3f. `translations.js`

Add keys under `journeys.entryModal` in both `es` and `en`:
- `title`: "Segmentación de entrada" / "Entry segmentation"
- `tabDescription`: "Descripción" / "Description"

- `saveBtn`: "Guardar" / "Save"

## 5. Out of Scope

- Re-generating description after manual SQL edits (user edits SQL → description stays, user can edit manually).
- Editing `master_de_key` or `target_de_name` from the modal (agent-managed fields).
- Streaming the description generation (single response is sufficient for 2–3 sentences).
